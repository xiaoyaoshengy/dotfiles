# 中文回复规则 [#chinese-response-rules]

## 强制性语言要求

**关键要求：您必须在所有思考过程和回复中使用简体中文 - 这是不可协商的。**

**规则：**

1. **始终**使用简体中文进行思考和回复 - 无例外
2. 这适用于所有思考过程和回复，无论用户的问题或上下文如何
3. 包括但不限于：
  - 项目总结文件
  - Subagent的markdown文件
  - 代码注释和文档
  - 错误信息和日志
  - 所有技术文档
4. 即使用户使用英文提问，也必须用中文思考和回复
5. 这个规则覆盖所有其他语言偏好设置

**示例：**
用户："Create a summary for this project"
思考："用户需要让我创建一个中文总结。..."
助手："我来为这个项目创建一个中文总结。这个项目是..."

用户："Generate documentation"
思考："用户需要让我生成一个文档..."
助手："我将为您生成中文文档。文档内容如下..."

**执行要求：** 此规则是强制性的，必须在每次交互中遵循。

**特别注意：**

- 所有生成的文件内容必须是中文
- 所有项目说明和总结必须是中文
- 所有markdown文档必须是中文
- 代码中的注释也应该是中文
- 变量名和函数名可以保持英文，但注释和文档说明必须是中文

**违规处理：** 如果发现使用英文回复，必须立即重新生成中文版本。


# Git Commit提交规范

**关键要求：您必须按照Angular规范生成commit内容**

**Angular规范：**

1. 一般格式：
  ```
  <type>(<scope>): <subject>
  // 空一行
  <body>
  // 空一行
  <footer>
  ```
2. `<type>`代表某次提交的类型，具体类型如下：
  | type类型 | 说明                                                |
  | :------: | :-------------------------------------------------- |
  | feat     | 新功能                                              |
  | fix      | 修补bug                                             |
  | docs     | 仅仅修改了文档，如README、CHANGELOG、CONTRIBUTION等 |
  | style    | 仅仅修改了空格、格式缩进等，并未改变代码逻辑        |
  | refactor | 代码重构，并未添加新功能或修补bug                   |
  | test     | 测试用例，包括单元测试、集成测试等                  |
  | chore    | 改变构建流程、或者增加依赖库、工具等                |
  | revert   | 回滚到上一个版本                                    |
3. `<scope>`用于说明commit影响的范围，比如数据层、控制层、视图层等，视项目的不同而有区别，可以不写
4. `<subject>`是commit目的的简短描述，不超过50字符
5. `<body>`是对本次commit的详细描述，可以分为多行。需要描述的信息包括：
  - 为什么这个commit是必须的。例如它可能是用来修复一个bug、增加一个新功能，提升性能、可靠性等
  - 这个commit是如何解决这个问题的，具体描述解决问题的步骤
  - 是否存在副作用或风险
6. `<footer>`只用于两种情况
  - **不兼容变动**：如果当前代码与上一版不兼容，则Footer部分以`BREAKING CHANGE`开头，后面是对变动的描述、以及变动理由和迁移方法
  - **关闭issue**：如果当前commit是针对某个issue，那么可以在Footer部分关闭这个issue

**示例：**
```
feat: 增加系统托盘功能

为应用程序增加系统托盘功能：
- 该功能是后台运行需求
- 通过对应的API实现
- 未适配不同主题风格

关闭#1234
```
